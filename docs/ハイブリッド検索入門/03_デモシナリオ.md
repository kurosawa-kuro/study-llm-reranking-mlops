# ハイブリッド検索デモシナリオ

**目的:** スライドで説明した `cache -> candidate search -> embedding -> rerank -> log` の流れを、API とコードで短く見せる  
**想定尺:** 4〜6 分  
**画面:** ターミナル + エディタ + API レスポンス

---

## デモ全体の流れ

1. `SearchParams` を確認して入力項目を把握する
2. `SearchPropertiesUseCase` を開いて主処理を読む
3. `curl /search` を実行してレスポンスを見る
4. `me5_score` と `search_log_id` / `compare_log_id` を確認する
5. `candidate_limit` を変えて挙動の違いを話す

---

## Step 1: 入力の確認

### 開くファイル
- `src/api/schemas.py`

### 見せる箇所
- `q`
- `city`
- `layout`
- `price_lte`
- `pet`
- `walk_min`
- `limit`
- `candidate_limit`

### ナレーション要点

> この API はフリーワードだけでなく、構造化条件も同時に受け取ります。  
> さらに、最終返却件数と候補件数が分かれている点が、再ランキング型の検索らしい設計です。

---

## Step 2: 主処理の確認

### 開くファイル
- `src/application/usecases/search_properties.py`

### 注目行
- `cache_port.get`
- `property_search_port.search_candidates`
- `embedding_port.embed_query`
- `reranking_port.rerank`
- `create_search_log`
- `create_compare_log`
- `cache_port.set`

### ナレーション要点

> 検索の主線はここです。  
> キャッシュを見て、候補を集めて、意味類似度を足し、LightGBM で並べ替え、最後にログを残します。  
> この順番を頭に入れておくと、後続のアダプタ実装が読みやすくなります。

---

## Step 3: API を叩く

### 例コマンド

```bash
curl --get 'http://localhost:8000/search' \
  --data-urlencode 'q=静かな住宅街 2LDK' \
  --data-urlencode 'city=札幌' \
  --data-urlencode 'pet=true' \
  --data-urlencode 'limit=5' \
  --data-urlencode 'candidate_limit=20'
```

### 画面で見るポイント
- `items`
- `count`
- `search_log_id`
- `compare_log_id`
- 各 item の `me5_score`

### ナレーション要点

> このレスポンスには結果一覧だけでなく、検索ログと比較ログの ID も含まれます。  
> つまり、検索 API の返却値がそのまま後続の分析導線になっています。

---

## Step 4: 再ランキングの中身を見る

### 開くファイル
- `src/adapters/outbound/ranking/lgbm_reranking_adapter.py`
- `src/services/ranking/lgbm_reranker.py`

### 見せる箇所
- `cosine_similarity`
- `item["me5_score"]`
- 特徴量ベクトル
- `model.predict(matrix)`
- フォールバックの weighted sum

### ナレーション要点

> ここでは、ME5 のスコアを直接順位にしているわけではありません。  
> `me5_score` は LightGBM に渡す特徴量の一つです。  
> 価格や CTR などと一緒に見て、最終順位を決めている点が重要です。

---

## Step 5: `candidate_limit` の話

### 追加コマンド例

```bash
curl --get 'http://localhost:8000/search' \
  --data-urlencode 'q=静かな住宅街 2LDK' \
  --data-urlencode 'city=札幌' \
  --data-urlencode 'limit=5' \
  --data-urlencode 'candidate_limit=100'
```

### 比較観点
- レスポンス時間
- 上位 5 件の並び
- `count` と候補の広さ

### ナレーション要点

> 候補件数を増やすと、より良い候補が後段に渡る可能性があります。  
> その代わり、DB 特徴量取得や再ランキングのコストは増えます。  
> この値は、精度と速度のバランスを見る代表的なチューニングポイントです。

---

## Step 6: 締め

### 伝えること
- ハイブリッド検索は 1 箇所で完結しない
- 候補取得と再ランキングを分けて考える
- ログまで含めて改善可能な設計になっている

### クロージング例

> このプロジェクトでは、検索エンジン、埋め込み、学習済みモデル、ログ保存を分けて実装しています。  
> そのため、どこを改善すれば品質が上がるかを切り分けやすい構造になっています。

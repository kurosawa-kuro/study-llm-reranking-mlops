承知しました。
現状の構成を見ると、機能はかなり揃っていますが、**命名軸が混在**しています。

具体的には、いまは次の軸が混ざっています。

* 技術名ベース：`ml`, `infra`, `api`
* 業務責務ベース：`ranking`, `search`, `evaluation`
* 実行形態ベース：`batch`
* 実装詳細ベース：`meili_client.py`, `train_lgbm.py`, `weekly_eval_report.py`

そのため、**「どこに何を置くか」の判断基準が少しぶれやすい状態**です。
実装カタログ上でも、`src/api`, `src/search`, `src/ranking`, `src/ml`, `src/batch`, `src/eval`, `src/infra` が並列に存在しており、さらに `batch` 配下だけ細かく整理されているため、階層ルールが統一され切っていません。

以下、**挙動非変更**を前提に、フォルダ名・ファイル名の整理案を提案します。

# 結論

一番きれいなのは、**「レイヤ + 業務責務」で統一**する形です。

現在のような：

```text
api / search / ranking / ml / batch / eval / infra
```

ではなく、

```text
presentation / application / domain / infrastructure / jobs
```

または、もう少し実務寄りに

```text
interfaces / usecases / services / repositories / jobs
```

へ寄せると、今後 Phase 7 以降も伸ばしやすいです。

# 命名の基本方針

## 1. 技術名ではなく責務名を優先

たとえば `ml/` は曖昧です。
その中に今あるのは `me5_embed.py`, `similarity.py`, `training_data.py`, `train_lgbm.py` ですが、実態は

* embedding生成
* 類似度計算
* 学習データ作成
* 学習実行

であり、全部が同じ粒度ではありません。

なので `ml` よりも、

* `embeddings`
* `training`
* `evaluation`

に分ける方が分かりやすいです。

---

## 2. “何をするファイルか” が名前で分かるようにする

たとえば `daily_stats.py` は少し広いです。
実装カタログ上の説明は「CTR 再集計 / 人気物件更新 / inactive 除外」です。

この場合、名前だけ見ると何の stats か曖昧なので、たとえば：

* `aggregate_daily_property_stats.py`
* `refresh_property_ranking_features.py`

の方が役割が読みやすいです。

---

## 3. report / eval / metrics を分ける

現状でも `batch/evaluation/metrics/` と `batch/evaluation/reports/` に分け始めていて、この方向は良いです。
ただし `offline_eval.py` と `offline_metrics.py` が別階層で共存していて、初見だと少し紛らわしいです。

なので、

* 計算ロジック → `*_service.py` or `*_calculator.py`
* 実行入口 → `run_*.py`
* 出力 → `export_*.py`

に寄せると衝突しにくいです。

# フォルダ設計で一番大事な修正点 3つ

## 1. `ml` を解体する

`ml` は責務が広すぎます。
embedding と training を分けた方がよいです。

## 2. `infra` を薄くする

今の `infra` は実質 repository + db + migrations です。
`repositories` と `migrations` を上げると理解しやすいです。

## 3. `batch` のファイル名を動詞始まりにする

実行入口であることが一目で分かります。

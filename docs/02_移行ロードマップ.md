# 不動産検索ランキング基盤
## 段階的コーディング移行ロードマップ

対象仕様: [01_仕様と設計.md](./01_仕様と設計.md)

---

## 0. 目的と方針

本ロードマップは、[01_仕様と設計.md](./01_仕様と設計.md) の内容を
「実装可能な順序」に分解し、PoC から運用可能な MVP まで段階的に進めるための計画です。

方針:

- まず検索基盤を動かす（Meilisearch + FastAPI）
- 次にデータ収集を整える（feedback, stats, logs）
- その後に意味特徴量（ME5）と学習ランキング（LightGBM）を導入
- 最後に評価・運用・クラウド展開へ進む

## 2. フェーズ別実装計画

### Phase 0: 開発基盤の初期化

目的:

- 仕様にある構成をローカルで再現し、開発を開始できる状態にする

実装タスク:

1. `src/` 配下を仕様どおりに作成（`api`, `search`, `ranking`, `ml`, `batch`, `infra`）
2. `docker-compose.yml` を作成（api, postgres, meilisearch, redis, pgadmin）
3. `.env.example` 作成（DB, Meili, Redis, pgAdmin, model path）
4. FastAPI ヘルスチェックエンドポイントを実装（`GET /health`）

完了条件（DoD）:

- `docker compose up -d` で全サービスが起動する
- API, pgAdmin, Meilisearch にブラウザアクセスできる

---

### Phase 1: 検索 MVP（Meilisearch + FastAPI）

目的:

- 最小の検索体験を成立させる

実装タスク:

1. `properties` テーブル作成（DDL + マイグレーション）
2. サンプル物件データ投入（seed）
3. Meilisearch インデックス作成と初回同期バッチ作成
4. `GET /search` 実装（クエリ受け取り -> Meili 検索 -> 上位返却）
5. フィルタ対応（city, layout, price_lte, pet, walk_min）

完了条件（DoD）:

- 条件検索で 100 件候補が取得できる
- API レスポンスが仕様フォーマットで返る

---

### Phase 2: フィードバックとログ収集

目的:

- ランキング改善に必要な教師データを蓄積できるようにする

実装タスク:

1. `search_logs` と `property_stats` テーブル作成
2. `POST /feedback` 実装（click / favorite / inquiry）
3. 検索時に `search_logs` 保存（query, result_ids, clicked_id）
4. stats 更新ロジック（impression, click, favorite, inquiry）実装

完了条件（DoD）:

- feedback API で行動ログが保存される
- `ctr`, `fav_rate`, `inquiry_rate` の基礎値が更新される

---

### Phase 3: 特徴量生成と日次バッチ

目的:

- Ranker 前段の特徴量基盤を安定化する

実装タスク:

1. Feature 生成モジュール実装（price, walk_min, age, area, photo_count, ctr, fav_rate, inquiry_rate）
2. 日次バッチ実装
	- CTR 再集計
	- 人気物件更新
	- 掲載終了物件の除外
3. バッチ実行ログ保存（成功/失敗/件数）

完了条件（DoD）:

- 日次バッチ 1 回で特徴量が更新される
- バッチ失敗時に原因追跡できるログが残る

---

### Phase 4: ME5 意味特徴量の導入

目的:

- キーワード一致だけでは拾えない意味近接をランキングに反映する

実装タスク:

1. ME5 モデルロード処理実装（推論プロセス分離を推奨）
2. 物件説明文 embedding の事前計算バッチ作成
3. 検索クエリ embedding のオンライン生成実装
4. cosine 類似度から `me5_score` 生成
5. `me5_score` を Feature に統合

実装結果:

- `src/ml/me5_embed.py` で ME5 埋め込み生成を実装（オフライン環境向け fallback あり）
- `src/batch/embeddings/me5_generate.py` で active 物件の embedding 事前計算を実装
- `GET /search` にクエリ embedding + cosine 類似度 + 再ランキングを実装
- `search_logs.me5_scores` と `property_features.me5_score` に保存導線を追加

完了条件（DoD）:

- 曖昧クエリ（例: 静かな住宅街）で `me5_score` が算出される
- Feature テーブルに `me5_score` が保存される

---

### Phase 5: LightGBM ランキング導入

目的:

- 候補 100 件を学習ベースで最終順位化する

実装タスク:

1. 学習データ作成（`query_id, property_id, features, label`）
2. ラベル定義実装（inquiry=3, favorite=2, click=1, no_action=0）
3. LightGBM 学習スクリプト実装（ランキング目的）
4. 学習済みモデル保存とバージョン管理
5. 推論モジュール実装（`score = model.predict(features)`）
6. `GET /search` に再ランキング処理を組み込み（上位20件返却）

完了条件（DoD）:

- 検索結果が LightGBM スコア順で返る
- ベースライン（Meili 順）と比較可能な評価ログが残る

---

### Phase 6: 評価と運用指標の可視化

目的:

- モデル改善の有効性を定量的に判断できるようにする

実装タスク:

1. オフライン評価実装（NDCG@10, MAP, Recall@20）
2. オンライン KPI 集計ジョブ（CTR, お気に入り率, 問い合わせ率, CVR）
3. 週次レポート出力（CSV/Markdown）
4. モデル更新判定ルール策定（閾値ベース）

完了条件（DoD）:

- 週次で評価レポートが自動出力される
- モデル採用/非採用を数値で判断できる

---

### Phase 7: デプロイと本番運用準備

目的:

- ローカル PoC からクラウド運用へ移行する

実装タスク:

1. Docker イメージ分割（api, batch）
2. Cloud Run デプロイ設定（または ECS/Fargate）
3. マネージド DB/キャッシュ接続設定
4. シークレット管理（DB パスワード、API キー）
5. リリース手順書と障害対応手順書の整備

完了条件（DoD）:

- 本番同等環境で `search` と `feedback` が稼働
- デプロイ手順が再現可能

## 4. 依存関係と着手順

着手順（厳守推奨）:

1. Phase 0 -> 1（検索 MVP）
2. Phase 2（ログ収集）
3. Phase 3（特徴量基盤）
4. Phase 4（ME5）
5. Phase 5（LightGBM）
6. Phase 6 -> 7（評価運用・展開）

理由:

- 学習ランキングはログと特徴量がないと成立しないため
- 先に検索と収集を動かす方が手戻りが少ないため

---

## 5. リスクと先回り対策

| リスク | 発生ポイント | 対策 |
| --- | --- | --- |
| 学習データ不足 | Phase 2-5 | まずルールベース併用で運用開始し、一定期間ログ蓄積後に学習比率を上げる |
| ME5 推論遅延 | Phase 4 | 物件 embedding 事前計算、クエリ側のみオンライン計算、キャッシュ導入 |
| 特徴量不整合 | Phase 3-5 | Feature スキーマ固定、学習時/推論時の同一変換関数を共通化 |
| KPI 改善が不明瞭 | Phase 6 | ベースライン比較と採用閾値を事前定義 |

## 一点結論

まずは「検索を動かす -> ログを貯める -> 特徴量を固める -> LGBM を載せる」の順に進めると、
最小コストで失敗しにくく、仕様どおりの成果物に最短到達できます。

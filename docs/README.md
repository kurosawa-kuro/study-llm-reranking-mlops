## 2. 権威順位 (矛盾したときの勝者)

```
02_移行ロードマップ.md  >  01_仕様と設計.md  >  README.md
```

`CLAUDE.md` と `03_実装カタログ.md` は上 3 者から派生する従属ドキュメント。3 者が互いに矛盾した場合は必ず `02_移行ロードマップ.md` を正として他を合わせ、User に flag する。

**具体例**: Vertex AI 不採用の決定は `02_移行ロードマップ.md` に書かれている。`README.md` や `01_仕様と設計.md` が Vertex AI を匂わせる表現で残っていたら、それは修正漏れ。

## 3. 更新ルール (変更種別 → 直すドキュメント)

| 変更 | 連動して直す |
|---|---|
| スコープの増減 (採用 / 不採用の切り替え) | `02_移行ロードマップ.md` → `01_仕様と設計.md` → `README.md` |
| 新しい GCP リソース / ファイル / BQ テーブル追加 | `03_実装カタログ.md` |
| 特徴量追加・変更 | `01_仕様と設計.md` + `03_実装カタログ.md` + `04_運用.md §7.1` |
| 運用コマンド / 閾値の変更 | `04_運用.md` + `Makefile` |
| CI / デプロイ経路の変更 | `03_実装カタログ.md` (Phase 7) + `README.md` のデプロイ表 |
| Claude Code 向けの制約変更 | `CLAUDE.md` |
| 環境構築手順の変更 (初期化手順・ツール変更) | `04_運用.md §1` + `Makefile (ops-bootstrap / help)` |

**原則**: 連動する複数ドキュメントは **同一 PR で直す**。別 PR に分けると drift の温床になる。過去に Vertex AI 採否で 3 者が分岐した事例がある。

## 4. 書き方ルール

- **日本語で書く** (LLM への指示を含む)。英単語はそのまま OK (例: `lifespan`, `Pub/Sub`)
- **節番号を本文に対応させる** (`§1`, `§1.2` 形式)。他ドキュメントからの相互参照を壊さない
- **コマンドは `make` ターゲット優先**。特に `ops-bootstrap` / `ops-daily` / `ops-weekly` / `verify-pipeline` を運用入口として使う。生 `gcloud` / `bq` / `terraform` は動的引数 (revision 名・イメージタグ・トークン値) が必要なときに限り使う
- **前提を伏せない** — 要求ツール / 認証 / 権限を必ず明記する。「暗黙にインストール済み前提」は書かない
- **識別子は固有名を使う** — プロジェクト ID (`mlops-dev-a`) / リージョン (`asia-northeast1`) / テーブル (`mlops.training_runs`) / トピック (`predictions`, `retrain-trigger`) を `<foo>` でぼかさない
- **実行順序に依存する手順は番号付きリスト or STEP** — 上から叩けば成立する形に保つ
- **推測で書かない**。コマンドを書いたら実際に叩いてみる

## 5. ドキュメントを追加する前に

1. 既存 6 ドキュメントの役割と被らないか確認。被るなら追加せず既存を拡張する
2. 追加時は本 `docs/README.md` の § 1 に 1 行追加し、`../README.md` の「ドキュメント」節にもリンクを追加
3. 権威順位 (§ 2) の位置付けを決める (従属 / 並列)
4. 更新契機 (§ 3) を表に追加する

## 6. メンテナンス観点

- `02_移行ロードマップ.md` の「本リポジトリの決定的仕様」という位置付けは絶対。ここを weaken する表現 (「暫定」「TBD」) を入れるなら必ず User に確認
- `03_実装カタログ.md` は放置すると実態と乖離する。コード変更時に Phase 表と「未着手」節を更新すること
- `04_運用.md §1` (環境構築) は **上から順に叩いて完走することが品質基準**。初期化手順を変更する場合は `make ops-bootstrap` の完走を実測で確認する

# 図解（Mermaid）

動画・スライドで使うハイブリッド検索の図です。  
Mermaid Live Editor か `mmdc` で PNG / SVG に書き出す想定です。

---

## Diagram 1: 全体アーキテクチャ

```mermaid
flowchart LR
    U[ユーザー] --> API[FastAPI /search]
    API --> C{Redis<br/>cache hit?}
    C -->|yes| R[検索結果を返却]
    C -->|no| M[Meilisearch<br/>候補取得]
    M --> E[ME5 query embedding]
    E --> S[cosine similarity<br/>me5_score]
    S --> L[LightGBM rerank]
    L --> LOG[search_logs<br/>ranking_compare_logs]
    LOG --> CACHE[Redis set]
    CACHE --> R

    DB1[(property_embeddings)]
    DB2[(property_features)]
    E -. query vector .-> S
    DB1 --> S
    DB2 --> L
```

**用途:** 最初の全体像説明。  
**強調点:** 候補取得と再ランキングが別段になっていること。

---

## Diagram 2: `/search` のシーケンス

```mermaid
sequenceDiagram
    actor User
    participant Route as FastAPI route
    participant Usecase as SearchPropertiesUseCase
    participant Cache as Redis
    participant Search as Meilisearch adapter
    participant Embed as ME5 adapter
    participant Rank as LightGBM adapter
    participant Log as Log repository

    User->>Route: GET /search?q=静かな住宅街&city=札幌
    Route->>Usecase: execute(SearchQuery)
    Usecase->>Cache: get(cache_key)
    alt cache hit
        Cache-->>Usecase: cached response
        Usecase-->>Route: response
    else cache miss
        Usecase->>Search: search_candidates(query)
        Search-->>Usecase: candidates
        Usecase->>Embed: embed_query(q)
        Embed-->>Usecase: query_vector
        Usecase->>Rank: rerank(query, candidates, query_vector)
        Rank-->>Usecase: reranked items
        Usecase->>Log: create_search_log(...)
        Usecase->>Log: create_compare_log(...)
        Usecase->>Cache: set(cache_key, response)
        Usecase-->>Route: response
    end
    Route-->>User: JSON response
```

**用途:** 実コードの主線を説明するとき。  
**強調点:** キャッシュヒット時とミス時の分岐。

## Diagram 3: 候補取得と再ランキングの役割分担

```mermaid
flowchart TB
    Q[検索条件 + q] --> P1[Meilisearch]
    P1 --> C1[候補100件]
    C1 --> P2[ME5 cosine similarity]
    P2 --> P3[LightGBM]
    P3 --> O[上位20件]

    F1[city / layout / price / pet / walk_min] --> P1
    F2[description の意味] --> P2
    F3[ctr / fav_rate / inquiry_rate / area / age] --> P3
```

**用途:** ハイブリッド検索の概念整理。  
**強調点:** 1 段目と 2 段目で見ている情報が違うこと。

## Diagram 4: Embedding の手順

```mermaid
flowchart LR
    P[properties] --> B[build_passage]
    B --> EP[encode_passages]
    EP --> DB[(property_embeddings)]

    Q[query q] --> EQ[encode_queries]
    EQ --> SIM[cosine similarity]
    DB --> SIM
    SIM --> M[me5_score]
    M --> RERANK[LightGBM rerank]
```

**用途:** Embedding を重点説明するスライド用。  
**強調点:** 物件側は事前計算、クエリ側はオンライン計算であること。

## Diagram 5: 特徴量の合流点

```mermaid
flowchart LR
    subgraph Candidate["候補物件ごとの情報"]
        A[price]
        B[walk_min]
        C[age]
        D[area]
        E[ctr]
        F[fav_rate]
        G[inquiry_rate]
        H[me5_score]
    end

    Candidate --> LGBM[LightGBM model]
    LGBM --> Score[lgbm_score]
    Score --> Sort[score desc]
    Sort --> Top[返却順]
```

**用途:** `lgbm_reranker.py` の説明。  
**強調点:** `me5_score` は単独ではなく特徴量群の一つ。

## Diagram 6: ログと改善ループ

```mermaid
flowchart LR
    Search[検索実行] --> SL[search_logs]
    Search --> CL[ranking_compare_logs]
    UserAction[click / favorite / inquiry] --> FB[feedback]
    SL --> Analysis[オフライン分析]
    CL --> Analysis
    FB --> Analysis
    Analysis --> Train[再学習 / 特徴量改善]
    Train --> Deploy[モデル更新]
    Deploy --> Search
```

**用途:** 学習システムとしての全体像。  
**強調点:** 検索基盤は返して終わりではなく、改善ループを持つ。

---

## Diagram 7: キャッシュキーの考え方

```mermaid
flowchart TB
    Params[q, city, layout, price_lte, pet, walk_min, limit, candidate_limit]
    Params --> Key[cache key 生成]
    Key --> Redis[(Redis)]
    Redis --> Hit[hit: 結果を即返却]
    Redis --> Miss[miss: 検索処理を実行]
```

**用途:** レイテンシとログの話をする補助図。  
**強調点:** `limit` と `candidate_limit` もキーに入る。

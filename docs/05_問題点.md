## P0: 致命的バグ

### P0-1 — 再ランクが候補100件ではなく上位20件にしか効いていない
> ✅ **修正済** (`src/api/routes/search.py`) — `hits[:candidate_limit]` で100件取得後に再ランク、`items[:limit]` は再ランク後に実施。

**ファイル**: `src/api/routes/search.py` L73

```python
hits = result.get("hits", [])[:limit]   # ← candidate_limit=100 で取得後に先に20件に絞る
```

**期待**: Meili から `candidate_limit`（デフォルト100件）を取得し、その100件を ME5 → LightGBM で再ランクしてから上位 `limit`（20件）を返す。  
**実際**: Meili の結果を先に `[:limit]` で20件に切り捨ててから再ランクしている。100件から最適な20件を選ぶという設計意図が完全に失われている。

**修正方針**:
```python
hits = result.get("hits", [])[:candidate_limit]   # 候補全件で再ランク
...
items = rerank_with_lgbm(items)
items = items[:limit]                              # 再ランク後に上位 limit 件を切る
```

### P0-2 — `cosine_similarity` がコサイン類似度になっていない
> ✅ **修正済** (`src/ml/similarity.py`) — `math.sqrt` でノルムを計算し正規化。次元不一致時は `0.0` を返す次元ガードを追加。

**ファイル**: `src/ml/similarity.py`

```python
def cosine_similarity(a: list[float], b: list[float]) -> float:
    size = min(len(a), len(b))
    return float(sum(a[i] * b[i] for i in range(size)))  # 分母（ノルム積）なし
```

**問題**: 正規化済みベクトル同士の内積はコサイン類似度に等しいが、関数名が実装と乖離している。  
ME5 本物モデル使用時は `normalize_embeddings=True` で正規化済みのため結果は正しい。  
しかし `fallback_embedding` も `normalize()` 済みなので動作するが、**混在時**（物件が fallback 128次元・クエリが本物 1024次元、またはその逆）は `min(128, 1024)=128` 次元のみで計算されるため、スコアが実質無意味になる。

### P1-2 — `ensure_latest_offline_eval()` が毎週再計算しない
> ✅ **修正済** (`src/batch/evaluation/reports/weekly_eval_report.py`) — `ensure_latest_offline_eval()` の早期リターンを廃止。毎回 `compute_offline_metrics()` を呼んで新規 INSERT する。

**ファイル**: `src/batch/evaluation/reports/weekly_eval_report.py`

```python
row = cur.fetchone()
if row is not None:
    return row           # ← 既存レポートが1件でもあれば即返す
```

**問題**: 週次レポートを何度実行しても、`offline_eval_reports` に1件でもレコードがある限り、最初に保存した古い評価が再利用される。新しいフィードバックが蓄積されても評価が更新されない。

**修正方針**: `ensure_latest_offline_eval()` を廃止し、`main()` から毎回 `compute_offline_metrics()` を呼んで新規保存する。

### P1-3 — DB コネクションプール未使用

**ファイル**: `src/infra/database/db.py`

```python
def get_db_connection() -> psycopg.Connection:
    return psycopg.connect(...)    # 毎回 TCP 接続を確立
```

**問題**: 全バッチ・全リクエストで毎回新規接続する。FastAPI が並行リクエストを処理すると PostgreSQL の接続上限（デフォルト100）に近づく。  
`psycopg_pool.ConnectionPool` または `asyncpg` + `asyncpg.create_pool` に切り替えることでリソース効率が改善する。

### P1-4 — ME5 モデルがロードされていない場合に fallback と本物の埋め込み次元が混在する

**ファイル**: `src/ml/me5_embed.py`

- 本物モデル（`multilingual-e5-large`）: 1024次元  
- `fallback_embedding`: 128次元（ハードコード）

**問題**: 物件 embedding が fallback（128次元）で保存された後に本物モデルが利用可能になっても、DB の `property_embeddings` には128次元のベクトルが残る。クエリ embedding は1024次元になるため `cosine_similarity` は128次元で計算される（P0-2 に関連）。

**修正方針**: `fallback_embedding` の `dim` を本物モデルと合わせる（1024）か、embedding 保存時にモデル名・次元数をチェックして不整合を検出する。

---

### P1-5 — LightGBM モデルが API 起動後は更新できない
> ✅ **修正済** (`src/ranking/inference.py`) — `_MODEL_AVAILABLE: bool` フラグを `_MODEL_MTIME: float | None` に置換。`path.stat().st_mtime` が変わった場合のみモデルを再ロードする。

**ファイル**: `src/ranking/inference.py`

```python
_MODEL_AVAILABLE = False

def _load_model():
    global _MODEL, _MODEL_AVAILABLE
    if _MODEL_AVAILABLE:
        return _MODEL          # 一度 True になると再ロードしない
    _MODEL_AVAILABLE = True    # モデルロード失敗でも True に設定
    ...
```

**問題1**: モデルファイルが存在しない状態で起動すると `_MODEL = None` かつ `_MODEL_AVAILABLE = True` が確定し、その後モデルファイルを置いても API 再起動なしに読み込まれない。  
**問題2**: 週次再学習でモデルが更新されても反映されない。

**修正方針**: ファイルの更新日時（`mtime`）を保存し、`mtime` が変わった場合は再ロードする。

## P2: 信頼性の問題

### P2-1 — `log_ranking_comparison()` の失敗が検索 500 エラーを引き起こす
> ✅ **修正済** (`src/api/routes/search.py`) — `log_search_and_increment_impressions()` / `log_ranking_comparison()` を `try/except Exception` で囲み、ログ失敗時は `None` を返して検索結果は継続して返す。

**ファイル**: `src/api/routes/search.py`

```python
compare_log_id = log_ranking_comparison(...)    # 例外が伝播すると /search が 500
```

**問題**: ログ記録の失敗がエンドユーザーへの検索エラーを引き起こすべきではない。同様に `log_search_and_increment_impressions()` も例外時に 500 になる。

**修正方針**:
```python
try:
    compare_log_id = log_ranking_comparison(...)
except Exception:
    compare_log_id = None   # ログ失敗は無視して検索結果は返す
```

### P2-2 — Meilisearch `add_documents` がインデックス反映完了を待たない

**ファイル**: `src/search/meili_client.py`, `src/batch/search_index/meili_sync.py`

Meilisearch はドキュメント追加をタスクキューで非同期処理する。`add_documents` が返った直後に検索しても新しいドキュメントが見えないことがある。

**修正方針**: `POST /indexes/{uid}/documents` のレスポンス `taskUid` を取得し、`GET /tasks/{taskUid}` でステータスが `succeeded` になるまでポーリングする。

### P2-3 — `property_stats` の impression 更新が N 回の個別 SQL で実行される
> ✅ **修正済** (`src/infra/repositories/engagement.py`) — `unnest(%s::bigint[])` を使ったバルク UPSERT に変更し、1 SQL で全件処理。

**ファイル**: `src/infra/repositories/engagement.py`

```python
for property_id in result_ids:       # 20件のループ
    cur.execute("INSERT INTO property_stats ... ON CONFLICT DO UPDATE ...", ...)
```

**問題**: 1 リクエストごとに最大 `limit`（デフォルト20）件の個別 SQL が発行される。高負荷時のパフォーマンスボトルネックになる。

**修正方針**: `executemany` またはバルク UPSERT（`UNNEST` + `JOIN`）に変更する。

## P3: 評価指標の品質問題

### P3-1 — 1 クエリにつき評価できるアクションが1件のみ

**ファイル**: `src/infra/migrations/006_phase5_learning_logs.sql`, `src/eval/offline_metrics.py`

`search_logs` テーブルの `actioned_id` / `action_type` は単一カラムのため、同じ検索セッションで複数回フィードバックされても最後の1件のみが保存される（`apply_feedback` で上書き）。

**影響**: オフライン評価が「最後のアクション」のみに基づく。正しい評価には複数の関連アイテムを記録できるスキーマが必要。

---

### P3-2 — NDCG / MAP の実装が1関連文書前提の近似

**ファイル**: `src/eval/offline_metrics.py`

```python
def _ndcg_at_10(rank: int | None, gain: float) -> float:
    dcg = gain / math.log2(rank + 1)
    idcg = gain / math.log2(2)    # ← IDCG が「関連文書が1位にある場合」固定
    return dcg / idcg
```

正確な NDCG@k は全関連文書の DCG リストを構築して IDCG を計算する必要があるが、関連文書が1件しかない前提で計算している。P3-1 の問題が解決されれば、この計算も修正が必要になる。

### P3-3 — CVR の定義が文書化されていない

**ファイル**: `src/batch/evaluation/metrics/kpi_daily.py`, `src/batch/evaluation/reports/weekly_eval_report.py`

```python
cvr = (inquiries / clicks) if clicks > 0 else 0.0
```

**問題**: CVR を「問い合わせ数 / クリック数」と定義しているが、一般的な CVR は「コンバージョン（問い合わせ）/ セッション（表示）」とする解釈もある。定義が `03_実装カタログ.md` にも文書化されていない。

## P4: セキュリティ

### P4-1 — DB パスワードのデフォルト値が弱い
> ✅ **修正済** (`src/infra/database/db.py`, `.env.example`, `docker-compose.yml`) — `_require_env("POSTGRES_PASSWORD")` で未設定時に `RuntimeError`。`.env.example` は `<change_me>` に変更。`docker-compose.yml` の `env_file` を `.env.example` → `.env` に修正。

**ファイル**: `src/infra/database/db.py`, `.env.example`

```python
password=os.getenv("POSTGRES_PASSWORD", "password")  # フォールバックが "password"
```

`.env.example` にも `POSTGRES_PASSWORD=password` が平文で記載されている。環境変数が未設定の場合にデフォルト値で接続できてしまう。  
（OWASP Top 10: A05 Security Misconfiguration）

**修正方針**: デフォルト値を廃止し、環境変数未設定時は `RuntimeError` を送出する。`.env.example` にはダミー値ではなくプレースホルダー（`POSTGRES_PASSWORD=<change_me>`）を使う。

## P5: コード品質

### P5-1 — `_embedder` グローバルの遅延初期化がスレッドセーフでない
> ✅ **修正済** (`src/ml/me5_embed.py`) — `threading.Lock` によるダブルチェックロッキングを実装。

**ファイル**: `src/ml/me5_embed.py`

```python
_embedder: Embedder | None = None

def get_embedder() -> Embedder:
    global _embedder
    if _embedder is None:            # チェックと設定がアトミックでない
        _embedder = Embedder(...)
```

FastAPI のスレッドプールで複数リクエストが同時に `get_embedder()` を呼ぶと、初回に2回 `Embedder.__init__` が実行される可能性がある（モデルを2回ロードする）。  
`threading.Lock` でガードするか、アプリ起動時（`lifespan`）に初期化する。

---

### P5-2 — ME5 再ランクのソート結果が常に LightGBM に上書きされる
> ✅ **修正済** (`src/api/routes/search.py`) — `apply_me5_rerank` 内の `items.sort(...)` を削除。ME5 はスコア付与のみ行い、ソートは LightGBM が一括して担う。

**ファイル**: `src/api/routes/search.py`

```python
items = apply_me5_rerank(q, items)   # ME5 スコアでソート
items = rerank_with_lgbm(items)      # LightGBM スコアで再ソート（ME5 ソートは消える）
```

ME5 によるソートは LightGBM が必ず上書きするため、中間ソートが無駄な処理になっている。ME5 スコアを特徴量として付与するだけでよく、ソートは LightGBM のみで行えばよい。

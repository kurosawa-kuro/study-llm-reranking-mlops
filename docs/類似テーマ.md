# LightGBM（LGBM）+ ME5 + Elasticsearch

# 不動産検索ランキング基盤 提案書

## （非スタートアップ向け / 精度評価可能 / 後日EC検索へ横展開可能）

---

## 1. 本資料の目的

不動産検索ドメインを題材に、以下を満たす検索・推薦・ランキング基盤を提案する。

* **Elasticsearch** による候補物件検索
* **ME5** による意味類似検索・意味特徴量生成
* **LightGBM（LGBM）** による最終ランキング最適化
* **CTR / CVR 系KPIで評価可能**
* **後日EC商品検索へ転用しやすい**

---

## 2. なぜ不動産検索なのか

不動産検索は、EC商品検索にかなり近い構造を持つ。

| 不動産検索 | EC商品検索     |
| ----- | ---------- |
| 物件一覧  | 商品一覧       |
| 家賃    | 価格         |
| 築年数   | 発売日 / 型落ち度 |
| 面積    | スペック       |
| 問い合わせ | 購入CV       |
| お気に入り | カート / 保存   |

つまり、

> **一覧検索・属性絞り込み・比較・CV導線が揃っており、商品検索に近い**

ため、実務でもポートフォリオでも扱いやすい。

---

## 3. 想定ユースケース

* 賃貸物件検索
* 売買物件検索
* 投資用不動産検索
* 法人向け社宅候補検索

---

## 4. 基本アーキテクチャ

```text
ユーザー検索
   ↓
Elasticsearch
（条件一致・全文検索・フィルタ検索で候補100件取得）
   ↓
ME5
（クエリ・物件説明文の埋め込み類似度計算）
   ↓
Feature生成
   ↓
LightGBM Ranker
   ↓
上位N件表示
   ↓
クリック / お気に入り / 問い合わせログ収集
   ↓
再学習
```

---

## 5. 各技術の役割

### Elasticsearch

担当領域：

* エリア絞り込み
* 駅徒歩
* 家賃帯
* 間取り
* 築年数
* 設備条件
* 全文検索
* ソート用候補抽出

つまり、**高速な候補検索エンジン**。

---

### ME5

担当領域：

* クエリと物件説明文の意味類似度
* 「駅近」「静かな環境」「広め」などの曖昧要求の意味マッチ
* 類似物件推薦
* ベクトル検索特徴量生成

つまり、**意味理解を補う埋め込み層**。
multilingual-e5 は多言語埋め込みモデルとして公開されており、検索・類似度タスク向けの用途が明示されています。([Hugging Face][1])

---

### LightGBM（LGBM）

担当領域：

* Elasticsearchの候補 + ME5類似度 + 行動ログ特徴量を統合
* クリックされやすい順
* お気に入りされやすい順
* 問い合わせされやすい順

に最適化する。

つまり、**最終順位を学習で詰める層**。

---

## 6. 不動産検索での処理イメージ

### 例

ユーザー検索：

> 札幌 2LDK 駅近 8万円 ペット可

### Elasticsearchで候補抽出

* 札幌市
* 2LDK
* 家賃8万円前後
* 徒歩10分以内
* ペット可

### ME5で意味補完

* 「駅近」⇔「徒歩5分」「好立地」
* 「広め」⇔ 面積広い / LDK広い
* 「住みやすい」⇔ 周辺設備説明や紹介文との意味近接

### LGBMで再ランキング

* 類似度
* 家賃妥当性
* 人気駅
* 写真品質
* 過去CTR
* 問い合わせ率

を総合して最終順を決める。

---

## 7. 特徴量設計

### A. 物件属性特徴量

* 家賃
* 管理費込み価格
* 築年数
* 面積
* 間取り
* 階数
* 駅徒歩分
* 写真枚数
* 設備数

### B. 地域特徴量

* 駅人気度
* エリア相場との差分
* 周辺スーパー距離
* 学校距離
* 病院距離

### C. 行動特徴量

* 物件CTR
* お気に入り率
* 問い合わせ率
* 滞在時間
* 直帰率

### D. ME5由来特徴量

* クエリ embedding × 物件説明文 embedding cosine類似度
* クエリ embedding × 設備説明文類似度
* クエリ embedding × 周辺説明文類似度
* 類似閲覧物件との embedding 距離

ここが、今回の修正ポイントです。
**Elasticsearchだけでなく、ME5の意味類似度を特徴量としてLGBMに渡す** のが肝です。

---

## 10. なぜFAQ / 社内ナレッジ検索より強いか

FAQや社内ナレッジ検索は、

* 正解文書が曖昧
* クリックが解決を意味しない
* 利用回数が少ない
* 教師信号が弱い

ため、LGBMでのランキング学習が弱くなりやすい。

不動産検索は逆に、

* 表示
* クリック
* 保存
* 問い合わせ
* 内見
* 成約

まで追えるため、**教師データが作りやすい**。

## 11. 後日EC商品検索へ横展開しやすい理由

不動産検索はECに近いが、比較サイトよりは少し個体差が強い。
それでも次の置換が自然にできる。

| 不動産   | EC商品       |
| ----- | ---------- |
| 物件    | 商品SKU      |
| 家賃    | 価格         |
| 築年数   | 発売日        |
| 面積    | 商品スペック     |
| 問い合わせ | 購入         |
| お気に入り | カート投入 / 保存 |

したがって、

> **LGBM + ME5 + Elasticsearch による不動産検索ランキング基盤を、後日EC商品検索に横展開可能**

という説明ができる。

# 今回増えるもの

RAGから不動産検索ランキングへ変えると、主に追加されるのはこの3点です。

## 1. ME5でクエリ・物件文の埋め込みを作る

これは、今の embedding 処理の延長です。

RAGでは

* 文書チャンク embedding

今回では

* 検索クエリ embedding
* 物件説明文 embedding

に変わるだけです。

---

## 2. LGBM用の学習データを作る

ここがRAGと一番違う点です。

例えば：

* query_id
* property_id
* 家賃
* 駅徒歩
* 面積
* ME5類似度
* 過去CTR
* お気に入り率
* 問い合わせ率
* label

こういう表を作って、
**queryごとの group を持つ ranking 学習データ** にする。

---

## 3. 評価指標をRAG評価からランキング評価に変える

今は例えば

* Recall@K
* MRR
* Exact Match
* ROUGE-L

を回していますが、今度は

* NDCG@10
* MAP
* Precision@K
* Recall@K
* CTR
* お気に入り率
* 問い合わせ率

へ寄せるだけです。

つまり、**評価基盤の思想そのものは既に持っている** ということです。

## Phase 2

その後、既存 repo の流儀に寄せます。

* `src/ingestion` 相当で物件取込
* `src/pipeline` 相当で学習パイプライン
* `scripts/eval` 相当でランキング評価

* 検索条件を受ける
* 候補抽出
* feature生成
* LGBMで再順位付け
* 結果返却

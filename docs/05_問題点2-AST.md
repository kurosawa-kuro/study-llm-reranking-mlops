## 3. ルール定義（現行）

### stage 1

対象:

- `src/domain/entities`
- `src/application/usecases`

概要:

- domain/entities は標準ライブラリ + `src.domain.*` のみ許可
- application/usecases は標準ライブラリ + `src.application.*` + `src.domain.*` + `src.ports.*` を許可

### stage 2

追加対象:

- `src/api/routes`

概要:

- 直接禁止: `src.repositories.*`, `src.clients.*`, `src.adapters.outbound.*`
- 許可: `src.adapters.inbound.*`（DI provider 経由）

### stage 3

追加対象:

- `src/ports`

概要:

- 直接禁止: `src.adapters.*`, `src.clients.*`, `src.repositories.*`, `src.api.*`, `fastapi`, `pydantic`
- 許可: 標準ライブラリ + `src.ports.*` + `src.domain.*` + `src.application.dtos.*`

### stage 4

追加対象:

- `src/jobs`
- `src/trainers`

概要:

- jobs:
  - 直接禁止: `src.api.*`, `src.application.*`, `src.adapters.inbound.*`, `src.ports.inbound.*`
  - 許可: 標準ライブラリ + `src.jobs.*` + `src.core.*` + `src.repositories.*` + `src.services.*` + `src.clients.*` + `src.trainers.*` + `psycopg.*`
- trainers:
  - 直接禁止: `src.api.*`, `src.application.*`, `src.adapters.*`, `src.ports.inbound.*`
  - 許可: 標準ライブラリ + `src.trainers.*` + `src.core.*` + `psycopg.*` + `numpy` + `lightgbm`

### stage 5

追加対象:

- `src/adapters/outbound`

概要:

- 直接禁止: `src.api.*`, `src.application.usecases.*`, `src.adapters.inbound.*`
- 許可: 標準ライブラリ + `src.adapters.outbound.*` + `src.ports.*` + `src.clients.*` + `src.repositories.*` + `src.services.*` + `src.core.*` + `redis`

## 5. 導入時に起きた論点と対応

論点:

- `src/api/routes/*` が `src.adapters.inbound.fastapi.dependencies` を import する構造は許容するべきか。

対応:

- `api/routes` ルールで `src.adapters.inbound.*` を許可
- ただし `src.adapters.outbound.*` は引き続き禁止

判断理由:

- route から outbound adapter 直参照を防ぎ、UseCase + DI 経由を維持するため。

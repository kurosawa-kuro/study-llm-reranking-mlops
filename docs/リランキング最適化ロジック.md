## 1. 目的

不動産検索を題材に、以下を同時に満たす検索ランキング基盤の実装指針を示す。

- 候補検索: Elasticsearch または代替検索エンジン
- 意味特徴量: multilingual-e5-large（ME5）
- 最終順位最適化: LightGBM Ranker
- 評価: オフライン指標 + オンラインKPI
- 横展開: EC商品検索への転用可能性

---

## 2. アーキテクチャ概要

```text
ユーザー検索
  -> 候補検索エンジン（Elasticsearch / Meilisearch 等）で候補取得
  -> ME5 で query/文書の意味類似度を特徴量化
  -> LightGBM Ranker で再ランキング
  -> 上位結果表示
  -> 行動ログ（表示・クリック・保存・問い合わせ）収集
  -> 定期再学習
```

---

## 3. 実装フロー（WSL + Docker -> Vertex）

### 3.1 ローカル（WSL + Docker）

1. 物件データ/検索ログ/ラベルデータ準備
2. ME5 embedding 生成
3. 類似度特徴量作成
4. LightGBM Ranker 学習
5. 推論確認

### 3.2 GCP（Vertex）

1. 学習コードを Docker 化
2. Artifact Registry へ push
3. Vertex Custom Training 実行
4. GCS にモデル/評価成果物保存
5. 必要なら Vertex Pipelines 化

---

## 4. 特徴量とラベル

### 4.1 特徴量カテゴリ

- 物件属性: 家賃、築年数、面積、駅徒歩、設備数、写真数
- 地域属性: 駅人気、相場乖離、周辺施設距離
- 行動属性: CTR、お気に入り率、問い合わせ率
- ME5由来: query-物件説明文 cosine 類似度など

### 4.2 ラベル例

- 問い合わせ = 3
- お気に入り = 2
- 詳細閲覧 = 1
- 無反応 = 0

LightGBM ranking では query 単位の group を必ず持たせる。

## 6. 検索エンジン代替技術（要約）

### 6.1 推奨順

1. Meilisearch: 軽量、導入容易、PoC向き
2. Typesense: typo 耐性・facet 強め
3. OpenSearch: 本番/企業向け
4. PostgreSQL FTS/pg_trgm: DB完結で運用簡素
5. Qdrant: ベクトル検索特化
6. Weaviate: ハイブリッド検索

### 6.2 現実解（短期成果重視）

FastAPI + PostgreSQL + Meilisearch + LightGBM + ME5 + Docker Compose

## 7. 類似テーマとの比較

RAG テーマとの違いは「検索して答える」ではなく「検索して並べる」点。

- 共通: embedding、検索基盤、評価、MLOps
- 差分: ranking 学習データ作成、query group 設計、ラベル設計

このため、既存の MLOps 資産がある環境では移行しやすい。

---

## 8. 横展開（不動産 -> EC）

概念対応は以下の通りで、転用しやすい。

- 物件 -> 商品SKU
- 家賃 -> 価格
- 問い合わせ -> 購入
- お気に入り -> カート/保存

したがって、LGBM + ME5 + 検索基盤の構成は EC 検索にも拡張可能。

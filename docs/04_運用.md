# 運用

本ドキュメントは、ローカル開発環境での定常運用・確認・障害切り分け手順をまとめる。

最終更新: 2026-04-19

---

## 1. 事前準備

1. `.env` を作成し、`POSTGRES_PASSWORD` を必ず設定する
2. API コンテナを起動する

```bash
make up
```

3. ヘルスチェックを確認する

```bash
make health
```

---

## 2. 初期セットアップ

初回構築または環境を作り直したときは以下を実行する。

```bash
make ops-bootstrap
```

`ops-bootstrap` は以下を順に実行する。

- `phase1-bootstrap`
- `phase2-bootstrap`
- `phase3-bootstrap`
- `phase4-bootstrap`
- `phase5-bootstrap`
- `phase6-bootstrap`

---

## 3. 定常運用

### 日次運用

```bash
make ops-daily
```

実行内容:

- `phase1-sync`: DB → Meilisearch 同期
- `phase3-daily`: `property_stats` / `property_features` 日次更新
- `phase4-generate-embeddings`: active 物件の embedding 更新
- `phase6-kpi-daily`: KPI 日次集計

### 週次運用

```bash
make ops-weekly
```

実行内容:

- `phase5-compare-check`: Meili と rerank の差分確認
- `phase6-offline-eval`: オフライン評価を保存
- `phase6-weekly-report`: 週次レポート出力と採用判定
- `phase6-weekly-retrain`: 採用判定に応じた再学習

---

## 6. トラブルシュート

### `Search backend unavailable` が返る

1. `make up` を再実行
2. `make logs` で `api` と `meilisearch` のログ確認
3. `MEILI_HTTP_ADDR` と `MEILI_INDEX` の環境変数を確認

### バッチで DB 接続エラー

1. `.env` の `POSTGRES_PASSWORD` 設定を確認
2. `make logs` で `postgres` の起動状態を確認
3. 必要なら `make down && make up` で再起動

### 学習が失敗する

1. `make phase5-label-seed` で教師データを追加
2. `make phase5-generate-train` を実行
3. `make phase5-train` を再実行

### 週次再学習がスキップされる

`phase6-weekly-retrain` は直近判定が `adopt_lgbm=False` の場合にスキップする。先に `phase6-offline-eval` と `phase6-weekly-report` の結果を確認する。

---

## 7. 運用ポリシー

- 既存 migration SQL は原則書き換えず、新しい番号で追加する
- 定常運用は `ops-daily` / `ops-weekly` を入口にし、個別ターゲットは障害切り分け時のみ使う
- ドキュメント更新時は `docs/03_実装カタログ.md` と整合を取る
